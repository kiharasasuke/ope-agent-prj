# GAS Webアプリ実装計画: 仕訳マイスター（改訂版）

## 概要

銀行取引明細PDFをGemini APIでOCR処理し、弥生会計インポート用CSV（25項目形式）を生成するGoogle Apps Script Webアプリ。ユーザーが仕訳ルールを管理でき、相手方勘定科目を柔軟に設定可能。

## 要件

| 項目 | 内容 |
|------|------|
| アプリ名 | 仕訳マイスター |
| 入力 | Googleドライブ内のPDF（銀行取引明細） |
| 対象店舗 | 呑家、西口アオギリ、ホドケバ、はんろく、警視鳥 |
| OCR | Gemini API (gemini-pro-vision) |
| 出力1 | Googleスプレッドシート（新規作成して処理結果表示） |
| 出力2 | 弥生会計形式CSV（Shift-JIS, CRLF） |
| UI | キムラ式デザイン（フォレストグリーン配色） |
| UI形式 | ルールベース設定（摘要パターン→勘定科目のルールを事前設定、新規パターンのみ確認） |
| マスター管理 | 専用Googleスプレッドシートで勘定科目・補助科目・仕訳ルールを管理 |
| OCR確認 | 信頼度フラグ（高🟢/中🟡/低🔴）を表示し、低信頼度行を強調 |

---

## 画面遷移フロー

```
[STEP 1] フォルダ選択
    ↓
[STEP 2] PDF一覧 + 科目設定（店舗名→デフォルト勘定科目・補助科目を表示、修正可能）
    ↓
[STEP 3] 日付範囲選択（カレンダー + 先々月/先月ボタン）
    ↓
[STEP 4] OCR解析実行（プログレス表示）
    ↓
[STEP 5] 解析結果確認（信頼度フラグ付き、低信頼度行ハイライト、新規パターンはルール設定ダイアログ）
    ↓
[STEP 6] 出力（スプレッドシート保存 + CSVダウンロード）
```

## 主要関数設計

### コード.gs（サーバーサイド）

| カテゴリ | 関数名 | 役割 |
|----------|--------|------|
| 初期化 | `doGet()` | Webアプリエントリーポイント |
| 初期化 | `initializeMasterData()` | マスターデータSS初回作成 |
| Drive | `getPickerOAuthToken()` | Google Picker用OAuthトークン取得 |
| Drive | `getPdfListFromFolder(folderId)` | フォルダ内PDF一覧取得 |
| Drive | `detectStoreFromFilename(filename)` | ファイル名から店舗自動判別 |
| マスター | `getAccountMaster()` | 勘定科目マスター取得 |
| マスター | `getSubAccountMaster(accountId)` | 補助科目マスター取得 |
| マスター | `getStoreMaster()` | 店舗マスター取得 |
| マスター | `getJournalRules()` | 仕訳ルール一覧取得 |
| マスター | `saveJournalRule(ruleData)` | 仕訳ルール保存 |
| マスター | `updateJournalRule(ruleId, ruleData)` | 仕訳ルール更新 |
| マスター | `deleteJournalRule(ruleId)` | 仕訳ルール削除 |
| OCR | `processSinglePdf(fileId, storeConfig)` | PDFのOCR処理 |
| OCR | `callGeminiApi(base64Image, prompt)` | Gemini API呼び出し |
| OCR | `parseOcrResult(ocrText, storeId)` | OCR結果パース |
| OCR | `calculateConfidenceScore(row, prevRow)` | 信頼度スコア算出 |
| 検証 | `validateTransactions(data)` | 残高整合性チェック |
| 検証 | `autoCorrectOcrError(row, expectedBalance)` | 1桁OCR誤読の自動補正 |
| 仕訳 | `applyJournalRules(transactions, storeId)` | 仕訳ルール適用 |
| 仕訳 | `matchJournalRule(description, amount, isDeposit, storeId)` | 単一取引のルールマッチング |
| 仕訳 | `convertToYayoiFormat(journalEntries)` | 弥生会計形式変換 |
| 出力 | `createOutputSpreadsheet(title)` | 出力用SS作成 |
| 出力 | `saveToSpreadsheet(ssId, data, sheetName)` | データをシートに保存 |
| 出力 | `generateYayoiCsv(yayoiData)` | 弥生CSV生成（Shift-JIS + CRLF） |
| ユーティリティ | `convertToShiftJis(text)` | UTF-8からShift-JIS変換 |
| ユーティリティ | `normalizeDate(dateStr)` | 日付正規化（元号→西暦） |

## ルールマッチングアルゴリズム

```
1. 全ルールを優先度降順でソート
2. 各取引に対して:
   a. 入出金区分が一致するルールをフィルタ
   b. 摘要パターンをマッチング:
      - 完全一致: description === pattern
      - 部分一致: description.includes(pattern)
      - 正規表現: new RegExp(pattern).test(description)
   c. 最初にマッチしたルールを採用
   d. マッチしない場合は「要設定」フラグ（信頼度「低」）
```

## 技術的注意点

### GASの制限
- 実行時間: 6分制限（長時間処理は分割必要）
- Shift-JIS: ネイティブ非対応（文字コード変換必要）

### Gemini API
- モデル: gemini-pro-vision（画像入力対応）
- 画像サイズ: Base64で4MB以下推奨
- APIキー: PropertiesServiceで管理

### PDF処理
- GASではPDFの直接画像変換が困難
- 解決策: クライアントサイドでPDF.jsを使用してCanvas変換

## 検証方法

1. **マスターデータ検証**: スプレッドシートの読み書きが正常に動作することを確認
2. **OCR精度検証**: テストPDFで期待値との差異確認、信頼度フラグの適切な付与
3. **ルールマッチング検証**: 既存パターン（AD、ATM出金、ｱｵｷﾞﾘｺｰﾎﾟﾚ等）が正しくマッチすることを確認
4. **新規ルール登録検証**: ダイアログからのルール登録がマスターに反映されることを確認
5. **弥生インポート検証**: 生成CSVを弥生会計で試験インポート（Shift-JIS + CRLF）
6. **UI検証**: テーマ切り替え、ステップ遷移、フィルター機能の動作確認
